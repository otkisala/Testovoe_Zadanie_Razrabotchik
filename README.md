# Интеллектуальный менеджер задач

Веб-приложение для управления задачами (To-Do List) с интегрированным LLM-ассистентом на базе **Ollama** (модель `llama3.2:3b`). Позволяет управлять задачами в полном CRUD-цикле и использовать AI-функции: автоматическую категоризацию, декомпозицию, предложение приоритета и сводку рабочей нагрузки.

## Реализованные функции

| ID | Функция | Описание |
|----|---------|----------|
| US-1 | CRUD задач | Создание, просмотр, редактирование, удаление задач с подтверждением |
| US-2 | Фильтрация и поиск | Фильтры по статусу, приоритету, сроку + полнотекстовый поиск (на стороне сервера) |
| US-3 | LLM: Умная категоризация | AI предлагает категорию и теги — автоматически при создании и по кнопке на карточке |
| US-4 | LLM: Декомпозиция задачи | AI разбивает задачу на 3–5 подзадач; каждую можно принять или пропустить |
| US-5 | LLM: Предложение приоритета | AI анализирует описание и срок, предлагает приоритет с объяснением |
| US-6 | LLM: Сводка нагрузки | AI генерирует естественно-языковую сводку с рекомендациями по всем задачам |

**Дополнительно реализовано:**
- SSE-стриминг ответов AI (токены отображаются в реальном времени)
- Кеш LLM-ответов с TTL 5 минут и индикатором `⚡ из кеша`
- Счётчик токенов во время генерации
- Автоматические SQL-миграции при старте сервера
- Версионирование API (`/api/v1/`)
- Few-shot промпты и валидация/коерция JSON-ответов модели

## 2. Настройка среды

### Требования

| Инструмент | Версия | Назначение |
|---|---|---|
| Node.js | >= 18 | Бэкенд и фронтенд |
| npm | >= 9 | Менеджер пакетов |
| Ollama | последняя | Локальный LLM-сервер |

### Установка зависимостей

```bash
# Бэкенд
cd backend
npm install

# Фронтенд
cd ../frontend
npm install
```

### Переменные окружения

Создайте файл `backend/.env` (все переменные опциональны — показаны значения по умолчанию):

```env
# Порт бэкенда
PORT=3000

# URL Ollama-сервера
# Важно: используйте 127.0.0.1, а не localhost —
# Node.js на Windows может резолвить localhost в ::1 (IPv6)
OLLAMA_URL=http://127.0.0.1:11434

# Модель Ollama
OLLAMA_MODEL=llama3.2:3b
```

### Установка и запуск Ollama

1. Скачайте и установите Ollama с [ollama.com](https://ollama.com)
2. (Опционально) Если хотите хранить модели на другом диске, установите переменную окружения `OLLAMA_MODELS` до запуска сервера:
   ```
   # Windows (PowerShell)
   $env:OLLAMA_MODELS = "D:\ollama-models"
   ollama serve
   ```
3. Скачайте модель:
   ```bash
   ollama pull llama3.2:3b
   ```
4. Убедитесь, что сервер запущен:
   ```bash
   ollama serve
   # В браузере: http://127.0.0.1:11434
   ```

> AI-функции работают только при запущенном Ollama. Остальной функционал (CRUD, фильтры) работает независимо.

## 3. Запуск приложения

### Режим разработки

```bash
# Терминал 1 — бэкенд (порт 3000)
cd backend
npm run dev

# Терминал 2 — фронтенд (порт 5173)
cd frontend
npm run dev
```

Откройте: [http://localhost:5173](http://localhost:5173)

Фронтенд проксирует `/api` запросы на `localhost:3000` через Vite proxy — CORS-настройка не требуется.

### Продакшн-сборка

```bash
# Сборка бэкенда
cd backend
npm run build
npm start          # запускает dist/app.js на порту 3000

# Сборка фронтенда
cd frontend
npm run build      # артефакты в frontend/dist/
```

### Продакшн-деплой (пример с Nginx)

Nginx проксирует:
- `/api/` → `localhost:3000` (бэкенд Express)
- `/` → `frontend/dist/` (статика React)

## 4. Архитектурные решения

### Структура проекта

```
/
├── backend/
│   ├── migrations/          # SQL-миграции (001_, 002_, ...)
│   ├── src/
│   │   ├── app.ts           # Express app, middleware, маршруты
│   │   ├── database/db.ts   # Синглтон БД + авто-миграции при старте
│   │   ├── models/          # TypeScript-типы и DTO
│   │   ├── repositories/    # Слой доступа к данным (SQL)
│   │   ├── controllers/     # HTTP-валидация, коды ответа
│   │   ├── routes/          # Маппинг URL → controller
│   │   ├── services/        # Бизнес-логика (llmService: промпты, кеш, стриминг)
│   │   └── middleware/      # errorHandler, notFoundHandler
│   └── data/tasks.db        # SQLite-файл (создаётся автоматически)
└── frontend/
    └── src/
        ├── components/      # React-компоненты (TaskCard, TaskForm, ...)
        ├── hooks/           # useAiStream, useTasks
        ├── services/        # api.ts — HTTP-клиент
        └── types/           # TypeScript-типы фронтенда
```

### Ключевые решения

**Слой данных — Repository pattern**
Весь SQL изолирован в `taskRepository.ts`. Контроллеры вызывают методы репозитория (`findAll`, `findById`, `create`, `update`, `delete`) и не содержат SQL-запросов. Миграции применяются автоматически при старте через таблицу `_migrations`.

**API — RESTful с версионированием**
Все эндпоинты под префиксом `/api/v1/`. Единый формат ответа: `{ success: true, data }` или `{ success: false, error: { message, code } }`. Это позволяет фронтенду обрабатывать ошибки единообразно независимо от эндпоинта.

**LLM-интеграция — SSE-стриминг**
Вместо polling или long-polling используется Server-Sent Events (SSE) через `POST`-эндпоинты `/api/v1/ai/stream/*`. Фронтенд читает поток через `fetch` + `ReadableStream.getReader()` (т.к. стандартный `EventSource` работает только с GET). AbortController привязан к событию `res.on('close')` — при закрытии страницы запрос к Ollama отменяется.

**LLM-кеш**
`LlmCache` — Map с TTL 5 минут. Ключ включает содержимое задачи (не только id), поэтому при изменении задачи кеш инвалидируется автоматически.

**Prompt Engineering**
Каждый промпт состоит из системного сообщения (`role: system`) с определением роли и ограничений, и пользовательского (`role: user`) с контекстом задачи и 1-shot примером. После получения ответа применяется валидация с коерцией типов (`validateCategorization`, `validateDecompose`, `validatePriority`, `validateWorkload`) — некорректные enum-значения заменяются на дефолтные, массивы нормализуются.

**Фронтенд — хук `useAiStream`**
Универсальный хук инкапсулирует логику SSE-стриминга: состояния `idle → streaming → done`, счётчик токенов, флаг `cached`, `AbortController` для отмены. Все AI-кнопки используют один хук.

## 5. Известные проблемы и ограничения

- **Ollama на Windows и localhost**: Node.js на Windows может резолвить `localhost` в `::1` (IPv6), тогда как Ollama слушает `127.0.0.1`. Используйте явный `OLLAMA_URL=http://127.0.0.1:11434`.
- **Размер модели**: `llama3.2:3b` занимает ~2 ГБ. На слабых машинах первый запрос может занять 10–30 секунд (модель загружается в память).
- **Качество ответов AI**: небольшая модель (3b) иногда возвращает неточные категории или приоритеты. Валидация на стороне сервера гарантирует корректную структуру, но не смысловое качество.
- **Аутентификация**: приложение не имеет системы пользователей — все задачи общие. Для многопользовательского режима потребуется добавить auth-слой.
- **SQLite**: подходит для одного пользователя и разработки. При высокой конкурентной нагрузке следует мигрировать на PostgreSQL.

## 6. Что добавить при наличии времени

- **Авторизация**: JWT + регистрация/логин, изоляция задач по пользователю
- **Drag & Drop**: kanban-доска с перетаскиванием карточек между статусами

- **Тёмная тема**: переключатель light/dark с сохранением в localStorage
- **Тесты**: unit-тесты для `taskRepository` и `llmService`, e2e через Playwright
- **Экспорт**: выгрузка задач в CSV/JSON
- **WebSocket**: real-time синхронизация при работе с нескольких вкладок
- **История задач**: аудит-лог изменений (кто/когда/что изменил)
- **Более мощная модель**: переключение на `llama3.1:8b` или внешние API (OpenAI, Anthropic) для лучшего качества AI-функций
